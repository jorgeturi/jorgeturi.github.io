Trabajo final de programación orientada a objetos.

  • ¿Qué realiza la app?

    La aplicación permite escanear y crear códigos QR, los cuales pueden persistir tanto de forma local en el dispositivo como en la nube mediante Firebase.
    Es posible ver información de cada uno, elegir que sea estatico o dinamico
    Diseñado y desarrollado por: Turina Jorge
    Asistente principal: ChatGPT

  • ¿En qué parte se usaron los conceptos de POO?

    Cada archivo contiene alguna clase como AuthService, GenerateQRPage, LogPage, QRestatico, QRdinamico, etc. Por ejemplo, QRManager gestiona los QRs que fueron instanciados como Estaticos o Dinamicos según elección del usuario.

    Las variables y métodos privados, como los controladores TextEditingController, están encapsulados dentro de las clases y no se pueden acceder directamente desde fuera. Esto asegura la integridad de los datos y simplifica la interacción. (Por ejemplo, acceder y manipular el texto que el usuario ingresa en una pantalla no se desea que sea visto por cualquier pantalla)

    En Flutter, se utiliza herencia para extender clases como StatefulWidget y StatelessWidget. Las clases GenerateQRPage y LogPage extienden de estas clases base para personalizar comportamientos mientras aprovechan las funcionalidades predeterminadas.
    Ejemplo de funcionalidades predeterminadas:
    - StatefulWidget proporciona la capacidad de gestionar el estado de un widget de manera dinámica. Esto permite que los widgets actualicen su apariencia cuando cambian los datos que los afectan. Por ejemplo, en GenerateQRPage, mostrando o no un campo adicional para el tiempo de expiración si el QR es dinámico.

    - StatelessWidget es útil cuando el estado del widget no cambia después de su creación. Un ejemplo podría ser un componente visual estático, como un logo, que no requiere actualizarse una vez que se ha cargado.

    Dart no permite la herencia múltiple, pero se pueden usar mixins para compartir comportamientos entre clases sin la complejidad de la herencia múltiple

    El polimorfismo se ve en los widgets de Flutter que implementan el método build(). Aunque cada widget (como Stateful o Stateless) tiene su propia versión del método, todos comparten la misma firma. En Dart no es posible sobrecargar funciones según el tipo de parámetros, pero es posible usar parámetros opcionales para corroborar en la función o el tipo dynamic.

    Las clases como AuthService y QRStorage proporcionan una abstracción de procesos complejos. Por ejemplo, AuthService maneja la autenticación con Firebase sin exponer la complejidad de sus operaciones internas al resto de la aplicación.

  • ¿Se utilizó algún patrón de diseño?

    Debido a la simplicidad de la aplicación, no se implementó un patrón de diseño complejo, ya que el objetivo era mantener la estructura liviana y fácil de entender. Sin embargo, se intentó aplicar el patrón Modelo-Vista-Controlador (MVC) en ciertas partes del proyecto para separar las responsabilidades y mejorar la organización del código.

    En este enfoque, la Vista se encarga de la interfaz de usuario, mientras que el Modelo maneja los datos y la lógica de negocio. El Controlador actúa como intermediario, gestionando la comunicación entre la Vista y el Modelo.

    Aunque la implementación completa de MVC no fue necesaria debido a la naturaleza sencilla de la app, se utilizó este enfoque en ciertas áreas clave para facilitar el mantenimiento y permitir una escalabilidad futura sin sobrecargar el diseño con complejidades innecesarias.

    Ventajas de este enfoque:

    • Desacoplamiento: Separación clara de las responsabilidades, lo que permite modificaciones sin afectar otras partes del sistema.
    • Mantenimiento simplificado: Facilita la actualización y el ajuste de cada componente sin necesidad de alterar la lógica global.
    • Escalabilidad: La estructura modular permite que la aplicación pueda evolucionar de manera más ordenada si se requiere agregar más funcionalidades.
